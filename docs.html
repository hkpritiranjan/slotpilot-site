Overview

What is SlotPilot?

SlotPilot is an automated clinic waitlist and no-show recovery system. When a booked patient cancels or doesn't show up, SlotPilot finds the next matching patient on the waitlist, texts them an offer, and books the slot automatically if they confirm — all within 15 minutes, with no staff involvement after the initial cancel action.

It is built as a self-hostable Next.js application for small clinics (physiotherapy, chiropractic, dental) that are losing significant revenue to unfilled cancelled appointments but don't have the budget for enterprise scheduling software.

  • SMS cascade: system texts waitlist patients one by one until someone confirms
  • 15-minute offer windows: each patient has 15 minutes to reply YES before the next is tried
  • 24h + 2h reminders: automatically sent to all booked patients to reduce no-show rates
  • Priority queue: high-priority patients jump the standard waitlist queue
  • Multi-practitioner: slots can be assigned to specific practitioners within a clinic
  • Mock SMS for local dev: full SMS lifecycle simulated in the UI, no external service needed

Architecture

SlotPilot is a standard Next.js 16 App Router application. All business logic lives in API route handlers (src/app/api/). The frontend is server-rendered React with client components only where interactivity is required.

src/
├── app/
│   ├── api/
│   │   ├── auth/         # login, signup, session, logout
│   │   ├── slots/        # CRUD + status transitions

│   │   ├── waitlist/     # Patient waitlist management
│   │   │   └── [id]/     # PATCH / DELETE single entry
│   │   ├── sms/          # SMS log + reply simulation

    ├── db.ts             # Prisma client singleton

Core Concepts

Slots

A slot represents a single appointment time in a clinic's schedule. Slots have the following statuses:

StatusMeaning
openSlot exists but no patient booked yet
bookedA patient is booked. 24h and 2h reminders will be sent.
cancelledPatient cancelled. Triggers the waitlist cascade immediately.
no_showPatient didn't show. Triggers the waitlist cascade for future slots.
completedAppointment occurred. Terminal state.
💡
The waitlist cascade is triggered the moment a slot transitions to cancelled or no_show status via the PATCH endpoint. This is the only action that triggers it.
Core Concepts

Waitlist

The waitlist is a per-clinic queue of patients waiting for a specific appointment type. Each entry has:

FieldTypeNotes
patientNamestringPatient's full name
phonestringMobile number (SMS target)
appointmentTypestringMust match one of the clinic's configured types
prioritynormal / highHigh priority patients are offered slots before normal-priority patients
preferredDaysstring[]Optional. Patient's preferred days (Mon–Sun). Not yet used in matching.
notesstringOptional. Staff notes on the patient.
statuswaiting / notified / booked / removedSet automatically by the system

Patients add themselves via the public join page at /join/[clinic-slug]. Staff can also add patients from the dashboard.

Core Concepts

Cascade Logic

When a slot is cancelled or no-showed, triggerWaitlistForSlot(slotId) runs:

1. Find all WaitlistEntry rows where:
   - clinicId matches the slot's clinic
   - appointmentType matches the slot's appointmentType
   - status = "waiting"
   Order by: priority DESC (high first), createdAt ASC (oldest first)

2. Take the first matching entry.

3. Create a Notification record:
   - status: "pending"
   - expiresAt: now + 15 minutes

4. Send SMS to the patient.

5. Update WaitlistEntry status to "notified".

When the patient replies:

YES reply → handleWaitlistReply(notificationId, "yes")
  - Check notification not expired
  - Update slot: status = "booked", patientName = waitlist patient
  - Update WaitlistEntry: status = "booked"
  - Update Notification: status = "accepted"
  - Send confirmation SMS
  - Schedule 24h and 2h reminders

NO reply → handleWaitlistReply(notificationId, "no")
  - Update Notification: status = "rejected"
  - Update WaitlistEntry: status = "waiting" (back in queue)
  - Call triggerWaitlistForSlot() again → next patient tried
⚠
If a notification expires (15 minutes, no reply), the next run of the cascade will skip this patient and try the next one. Expiry is checked at reply time, not on a cron — add a cron job in production to auto-expire and cascade forward.
Core Concepts

Reminders

When a slot is booked (either directly or via waitlist fill), the system creates two Reminder records: a 24h reminder and a 2h reminder. These are not sent immediately — they’re sent when POST /api/reminders is called.

In production, call this endpoint on a cron schedule (e.g. every 30 minutes via Vercel Cron Jobs or an external cron service). It processes all reminders where the scheduled time has passed but sentAt is null.

# Example: cron every 30 minutes via Vercel Cron
# vercel.json
{
  "crons": [
    {
      "path": "/api/reminders",
      "schedule": "*/30 * * * *"
    }
  ]
}
Core Concepts

SMS System

All SMS sending goes through src/lib/sms.ts, which is an abstraction layer. In local development (no SMS_API_KEY set), it writes messages to the SmsMessage database table instead of sending.

To send real SMS, replace the mock implementation with a Plivo or Twilio API call. The function signature is:

sendSms({
  clinicId: string,
  to: string,       // e.g. "+917400000000"
  body: string,     // message text
  tag: string       // label for the SMS log (e.g. "offer", "reminder", "confirmation")
}): Promise<void>

The SMS inbox (/dashboard/sms) shows all outbound messages and lets you simulate YES/NO replies during development.

💬
WhatsApp Business API integration via MSG91 or Gupshup is planned as the primary delivery channel for Indian clinics. The abstraction layer in sms.ts is designed to swap cleanly.

Deployment

Production Database

SlotPilot uses Prisma 7 with the libSQL driver adapter. For production, replace the local SQLite file with Turso (free tier: 500MB, 1B row reads/month):

# 1. Install Turso CLI
curl -sSfL https://get.tur.so/install.sh | bash

# 2. Create a database
turso db create slotpilot-prod

# 3. Get the URL and token
turso db show slotpilot-prod --url
turso db tokens create slotpilot-prod

# 4. Set in Vercel environment
DATABASE_URL=libsql://slotpilot-prod-username.turso.io
DATABASE_AUTH_TOKEN=your-auth-token

Update src/lib/db.ts to use the auth token when it’s set:

const adapter = new PrismaLibSql({
  url: process.env.DATABASE_URL!,
  authToken: process.env.DATABASE_AUTH_TOKEN,  // undefined for local, set for Turso
});
Deployment

Real SMS with Plivo or Twilio

Replace the mock in src/lib/sms.ts with a real API call. Example with Plivo:

import Plivo from "plivo";

export async function sendSms({ to, body, clinicId, tag }: SmsParams) {
  // Log to DB first (for SMS inbox)
  await db.smsMessage.create({ data: { clinicId, to, body, direction: "out", tag } });

  if (process.env.SMS_API_KEY) {
    const client = new Plivo.Client(process.env.PLIVO_AUTH_ID, process.env.SMS_API_KEY);
    await client.messages.create(process.env.SMS_FROM!, to, body);
  }
}

Set environment variables: PLIVO_AUTH_ID, SMS_API_KEY, SMS_FROM.

💬
For India, Plivo's India DLT registration is required for transactional SMS. Alternatively, MSG91 or Gupshup offer WhatsApp Business API which bypasses DLT for non-promotional messages.

FAQ

Does SlotPilot replace Practo?

No. Practo brings new patients via discovery. SlotPilot fills gaps from existing patients who are already waiting. They solve different problems and work together.

What happens if no waitlist patient accepts?

The cascade tries each patient in priority order. If everyone declines or the list is exhausted, the slot remains open — nothing bad happens. You can re-open it or manually book it.

Can I have multiple appointment types?

Yes. Configure as many as you need in clinic settings (e.g. "Initial Assessment, Follow-up, Dry Needling, Sports Massage"). Each type has its own independent waitlist queue.

Does the patient need to download an app?

No. Patients join via a web link, reply by SMS (or WhatsApp soon), and never need to create an account or install anything.

Is the waitlist position first-come-first-served?

Within the same priority tier (normal / high), yes — oldest join date is first. High-priority patients always come before normal-priority patients.

How do I handle multiple practitioners?

Each slot can be assigned to a practitioner. You can filter slots by practitioner in the dashboard. Waitlist matching is currently by appointment type only — practitioner filtering in the waitlist is on the roadmap.

Is this HIPAA compliant?

The local prototype is not designed for US HIPAA compliance. For US deployment, use AWS with a BAA, HIPAA-eligible Twilio/Plivo plans, and end-to-end encryption. For India, there is no equivalent mandatory standard yet, but the app stores minimal patient data (name + phone + appointment type).
